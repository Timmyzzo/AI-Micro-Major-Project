# 14 术语、决策记录与待确认项

## 1. 术语表

| 术语 | 含义 |
| --- | --- |
| 有功功率 | 实际做功的功率，本项目 Global_active_power 单位为 kW |
| 无功功率 | 在交流系统中往返交换的功率分量，数据字段单位标为 kW，沿用 UCI 描述 |
| 电量 | 一段时间消耗的能量，常用 Wh 或 kWh |
| 分项计量 | 特定区域或设备组的用电量；三个字段为每分钟 Wh 记录 |
| 未分项电量 | 总有功电量减三个分项计量的剩余部分 |
| 时间粒度 | 相邻数据点代表的时间间隔，本项目模型为 15 分钟 |
| 上下文长度 | 模型用于预测的历史点数，默认 672 |
| 预测长度 | 一次输出的未来点数，默认 96 |
| 多步预测 | 一次或递归预测多个未来时间点 |
| Patch | 把连续若干时间点组成一个 token 的片段 |
| PatchTST | 基于 patch 的时序 Transformer 模型 |
| 基线 | 用于判断复杂模型是否有价值的简单方法 |
| MAE | 平均绝对误差，单位与目标相同 |
| RMSE | 均方根误差，对大误差更敏感 |
| WAPE | 总绝对误差与总真实值之比 |
| sMAPE | 对称平均绝对百分比误差 |
| 共形预测 | 利用校准残差构建经验覆盖区间的方法 |
| PICP | 真实值落入预测区间的比例 |
| MPIW | 预测区间平均宽度 |
| 残差 | 真实值减预测值 |
| 异常分数 | 将偏离程度标准化后的数值 |
| 数据泄漏 | 训练或选模过程中使用了当时不应知道的未来信息 |
| 回测 | 在历史时间点模拟当时进行预测并与后续真实值比较 |
| 模拟实时回放 | 按时间推进历史 CSV，不是真实传感器流 |
| 情景模拟 | 在人为参数和约束下计算假设结果，不代表已执行 |
| 可转移负荷 | 可以在不改变主要用途的前提下调整时间的用电 |
| Evidence | 供报告引用的结构化计算事实 |
| evidence_id | 每条证据的稳定编号 |
| OpenAI 兼容接口 | 在 base URL、鉴权和请求格式上兼容 OpenAI SDK 的第三方服务 |
| LLM 回退 | API 不可用时由本地模板生成报告 |
| dataset_id | 数据集版本 ID |
| run_id | 一次处理或训练运行 ID |
| model_id | 已注册模型 ID |
| schema_version | 数据或对象契约版本 |
| MVP | 满足课程核心验收的最小可行版本 |

## 2. 架构决策记录

### ADR-001 选择实验 6 作为基础

- 状态：已决定
- 日期：2026-07-21
- 决策：使用家庭用电数据探索实验。
- 原因：数据易获取、26 万分钟记录训练量充足、字段丰富、RTX 4060 可处理、适合分析预测预警一体化。
- 代价：只有一个家庭和六个月，泛化有限。

### ADR-002 使用模块化 Streamlit 单体

- 状态：已决定
- 决策：不用独立前后端和微服务。
- 原因：课程项目、单机单用户、工作量小、演示部署简单。
- 代价：不适合高并发和复杂权限，但不在范围。

### ADR-003 模型粒度为 15 分钟

- 状态：已决定
- 决策：分钟数据先治理，再聚合为 15 分钟用于主模型。
- 原因：保留日内规律，同时降低序列长度、训练和绘图成本。
- 代价：无法识别部分分钟级短事件；原始分钟数据仍用于质量和细节分析。

### ADR-004 按月份切分

- 状态：已决定
- 决策：1 至 4 月训练、5 月验证、6 月测试。
- 原因：清晰、可复现、避免随机泄漏。
- 代价：训练集末尾有长缺失，需要剔除跨段窗口。

### ADR-005 PatchTST 为主模型

- 状态：已决定
- 决策：使用小型单变量 PatchTST，基线含季节朴素、Ridge 和 LSTM。
- 原因：技术较新、长上下文、可在本地 GPU 运行、实现成本可控。
- 代价：不保证优于简单基线，必须实测。

### ADR-006 使用共形预测区间

- 状态：已决定
- 决策：用验证残差按预测步长校准 90% 区间。
- 原因：实现简单、结果可解释、覆盖率可测试，并可服务预警。
- 代价：时间依赖下只有经验覆盖，不是严格保证。

### ADR-007 预警由确定性规则生成

- 状态：已决定
- 决策：质量、稳健规则和残差分数决定等级。
- 原因：可重复、可测试、有证据。
- 代价：需要校准阈值；不做真实故障诊断。

### ADR-008 不在 MVP 使用强化学习

- 状态：已决定
- 决策：用能量守恒的负荷转移情景代替 RL。
- 原因：数据无动作、奖励和环境反馈；RL 会显著增加工作量且难验证。
- 代价：创新点主要由 PatchTST、共形和 LLM 提供。

### ADR-009 大模型只做解释层

- 状态：已决定
- 决策：LLM 只接收结构化证据，不计算或修改数值。
- 原因：降低幻觉、隐私和不可复现风险。
- 代价：报告自由度受到限制，这是有意的安全边界。

### ADR-010 文件保存时序、SQLite 保存元数据

- 状态：已决定
- 决策：CSV/Parquet/权重存文件，SQLite 存索引和小记录。
- 原因：简单、无服务、适合本地。
- 代价：不面向并发；不在 MVP 需求。

### ADR-011 训练与演示分离

- 状态：已决定
- 决策：命令行脚本训练，Streamlit 只加载和推理。
- 原因：减少页面阻塞和现场不确定性。
- 代价：界面不能“一键正式训练”，但可展示训练结果。

### ADR-012 OpenAI 接口通过适配层

- 状态：已决定
- 决策：支持配置 base URL、模型和接口风格；第三方服务以 Chat Completions 为最低兼容基线，运行时按配置探测 Chat Completions 或 Responses。
- 原因：用户提供兼容接口，具体服务尚未知。
- 代价：需写一层小适配和连接测试。

### ADR-013 原始 CSV 只读

- 状态：已决定
- 决策：不移动、不覆盖 docs 中课程文件；处理产物独立保存。
- 原因：可审计、可复现、防止误改。
- 代价：可能保留一份受控副本，占用很小。

### ADR-014 使用 uv 锁定 Windows GPU 环境

- 状态：已决定并验证
- 日期：2026-07-21
- 决策：使用 uv 管理 CPython 3.11.14、`.venv` 和 `uv.lock`；通过显式 PyTorch 官方 CUDA 13.0 索引安装 `torch 2.13.0+cu130`。
- 原因：本机没有系统 Python 3.11，uv 可以在不改变系统默认解释器的情况下建立项目隔离环境，并把 GPU wheel 来源写入可审计锁文件。
- 代价：当前锁文件以 Windows 为目标环境；迁移到 Linux 或 macOS 时需要重新解析并验证平台 wheel。

## 5. 触发重新决策的条件

- 数据文件改变或增加新家庭；
- 预测目标或时间粒度改变；
- GPU 实际资源低于低内存档；
- 课程强制要求前后端分离或强化学习；
- 用户 API 不兼容 SDK 且无简单适配；
- PatchTST 严重劣于全部基线；
- 截止时间缩短；
- 课程要求真实流式数据；
- 数据许可限制项目提交。

触发后不要静默改变实现，应新增或更新 ADR。

## 8. 课程答辩常见问题建议

### 为什么不继续用 Prophet？

基础实验的随机抽样不适合严格时间序列多步评估。项目需要未来 24 小时直接预测、与深度基线比较和误差区间，因此使用 PatchTST；Prophet 可作为原实验背景，不作为主模型。

### 为什么不把缺失值填 0？

0 表示真实低负荷，缺失表示不知道。把长缺失填 0 会制造虚假谷值，影响训练、指标和预警。

### 为什么用 15 分钟？

在日内规律和本地资源之间折中。六个月仍有约 1.7 万点，可生成大量窗口，且 7 天上下文对 4060 更友好。

### 为什么没有强化学习？

没有动作、奖励和环境反馈，RL 结果难以验证。项目选择有能量守恒约束的可解释情景模拟，同时已使用时序 Transformer、共形预测和大模型。

### 大模型如何防止胡说？

它只收到结构化证据，输出必须引用 evidence_id 并通过 schema 和数值校验；失败或无证据时回退模板。

### 系统是实时的吗？

MVP 用历史 CSV 做模拟实时回放，明确不宣称连接真实传感器。架构为未来数据适配预留服务边界，但不在当前范围。

### 模型一定更好吗？

不预设。项目用固定测试集和基线比较；如果 PatchTST 没有优势，就如实报告并使用表现最佳的模型。
