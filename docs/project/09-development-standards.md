# 09 开发规范

## 1. 目标

本规范保证后续代码简单、可测试、可追溯，并避免课程项目常见问题：页面堆满业务逻辑、时序数据泄漏、密钥提交、指标不可复现、为了展示引入过度架构。

## 2. 基本原则

1. 一个变更对应一个明确需求编号。
2. 优先使用最少代码完成既定功能。
3. 不重构与当前需求无关的代码。
4. 时间序列正确性高于模型复杂度。
5. 确定性计算与大模型文本严格分离。
6. 所有外部输入均需校验。
7. 原始数据不可变。
8. 模型、配置和结果可追踪。
9. 失败应可理解、可恢复。
10. 文档和实现同步。

## 3. Python 规范

### 3.1 版本和风格

- Python 3.11；
- 遵循 PEP 8；
- Ruff 负责检查和格式；
- 建议行宽 100；
- 使用 4 空格缩进；
- 文件 UTF-8；
- 不使用通配符导入；
- 不在模块导入时执行训练、读大文件或调用 API。

### 3.2 类型

- 公共函数和方法必须标注参数及返回类型；
- 领域对象使用 Pydantic model 或 dataclass；
- 避免用无结构字典贯穿系统；
- 对可能缺失值使用明确 Optional 或联合类型；
- 不以 Any 绕过领域契约；
- DataFrame 边界要有列 schema 校验。

### 3.3 命名

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 模块和文件 | snake_case | data_validation.py |
| 函数和变量 | snake_case | build_forecast_request |
| 类 | PascalCase | ForecastService |
| 常量 | UPPER_SNAKE_CASE | DEFAULT_CONTEXT_LENGTH |
| 内部字段 | 英文稳定名 | global_active_power_kw |
| 页面标题 | 中文 | 负荷预测 |
| ID | 规定前缀 | fcst_、mdl_、alt_ |

名称应体现单位。功率字段用 _kw，电量用 _wh 或 _kwh，电压用 _v，电流用 _a。

### 3.4 函数

- 单一职责；
- 尽量无隐藏全局状态；
- I/O 与纯计算分开；
- 复杂条件拆成命名清晰的私有函数；
- 不吞掉异常；
- 默认参数不得是可变对象；
- 时间范围边界是闭区间还是半开区间必须写明；
- 数值函数明确 NaN、空数组和除零行为。

### 3.5 文档字符串

公共函数说明：

- 目的；
- 参数和单位；
- 返回；
- 异常；
- 时间边界；
- 是否修改输入；
- 数据泄漏约束，若适用。

简单私有函数无需冗长文档，但命名应自解释。

## 4. 模块边界

### 4.1 页面层

页面可以：

- 收集输入；
- 调用应用服务；
- 管理当前会话；
- 渲染结果和状态。

页面不可以：

- 直接处理 26 万行清洗；
- 写模型训练循环；
- 直接拼接 SQL；
- 直接调用 OpenAI SDK；
- 自行计算预警等级；
- 保存 API Key。

### 4.2 领域层

领域层：

- 不导入 Streamlit；
- 使用明确输入和输出对象；
- 尽量纯计算；
- 可被 pytest 单独调用；
- 不依赖当前工作目录；
- 不读取未经传入的隐式全局配置。

### 4.3 基础设施层

文件、数据库、外部 API 和日志均通过适配器访问。测试使用临时目录、内存 SQLite 或模拟客户端。

## 5. 配置规范

- 默认配置集中在 configs；
- 环境变量只处理环境差异和秘密；
- 用户会话设置不直接改默认文件；
- 配置启动时由 Pydantic 校验；
- 模型参数不散落在脚本；
- 每次运行保存解析后的非敏感快照；
- 所有阈值带单位和来源；
- 改变默认值需更新需求、测试或模型文档。

禁止：

- 在源码中写绝对用户路径；
- 在源码中写真实 Key；
- 用未记录的 notebook 单元参数作为最终训练配置；
- 页面和脚本各维护一套默认值。

## 6. 数据开发规范

1. docs 中的原始 CSV 只读。
2. 日期明确 day-first。
3. 问号转 NaN，不转 0。
4. 长缺失段不跨段插值。
5. 时间聚合明确 mean、sum 和有效比例。
6. 时间切分先于缩放器拟合。
7. 任何窗口不能跨 split 边界。
8. 训练、验证、测试边界写入 manifest。
9. 派生字段必须记录公式和单位。
10. 处理数据必须保留 missing_mask 或质量标记。

处理规则变化视为数据 schema 或 preprocessing 版本变化。

## 7. 模型开发规范

- 每个模型在同一数据和窗口上评估；
- 先运行朴素基线；
- 训练过程有固定种子；
- 早停只看验证集；
- 最终测试只在模型冻结后执行；
- 指标在原始 kW 单位计算；
- 保存最佳权重，不只保存最后权重；
- 保存失败状态，不把中断运行标记成功；
- 模型文件附哈希；
- 模型卡记录适用范围和限制；
- 训练日志不得只保留最好一次，失败和差结果也应可追踪。

不得为了让 PatchTST 看起来更好而给基线更少历史或不同测试区间。

## 8. 大模型开发规范

### 8.1 输入

- 只发送结构化证据；
- 默认不发送完整时序；
- Key 与提示词分离；
- 用户文本作为数据转义；
- 发送前可预览；
- 记录模板版本和 evidence_hash。

### 8.2 输出

- 要求结构化 schema；
- 使用 Pydantic 校验；
- 结论引用 evidence_id；
- 数值必须与证据完全一致；
- 无证据时拒绝推断；
- 不允许模型修改告警等级和模拟结果；
- 响应失败回退模板。

### 8.3 测试

- 自动测试使用 mock，不调用真实付费 API；
- 真实连接测试由用户显式触发；
- 测试超时、限流、错误 JSON、空响应和错误数值；
- 日志不包含 Authorization。

## 9. UI 开发规范

- 页面遵循[功能与界面规格](03-feature-and-ui-specification.md)；
- 业务逻辑通过服务调用；
- 长操作显示进度；
- 所有图表标单位和时间范围；
- 无数据不显示 0；
- 缺失区间不连成 0；
- 缓存结果有标识；
- 禁用状态说明原因；
- 页面刷新不重复调用 LLM；
- 展示模式与开发模式使用同一业务逻辑；
- 1920×1080 下进行人工检查。

## 10. 异常与错误处理

### 10.1 原则

- 在能补充上下文的边界捕获；
- 保留原始异常链；
- 用户信息简洁、可操作；
- 技术信息进入脱敏日志；
- 可恢复错误不终止整个应用；
- 不用裸 except；
- 不静默返回空结果。

### 10.2 错误对象

统一包含：

- code；
- user_message；
- technical_message；
- recoverable；
- suggested_action；
- request_id；
- cause，可选。

### 10.3 重试

- 本地确定性计算一般不自动重试；
- SQLite 锁定可短暂有限重试；
- LLM 超时、限流和 5xx 最多一次；
- 鉴权、配置和 schema 错误不重试；
- 所有重试有上限和日志。

## 11. 日志规范

### 11.1 级别

- DEBUG：开发诊断，不在展示默认开启；
- INFO：关键流程开始、结束和缓存命中；
- WARNING：可恢复质量问题或降级；
- ERROR：当前操作失败；
- CRITICAL：核心存储或数据完整性风险。

### 11.2 结构字段

- timestamp；
- level；
- event；
- request_id；
- dataset_id；
- run_id / model_id / forecast_id；
- duration_ms；
- status；
- error_code；
- safe_context。

### 11.3 禁止记录

- API Key；
- Authorization；
- 完整大模型请求；
- 用户上传文件完整内容；
- 不必要的绝对用户路径；
- 可能含个人信息的原始备注；
- 未脱敏异常对象。

## 12. SQLite 与文件规范

- 通过仓储层访问 SQLite；
- 使用参数化查询；
- 开启外键；
- 文件写入采用临时文件后原子替换；
- 产物目录按 ID 分隔；
- 保存前校验目标位于允许目录；
- 不覆盖不同 run_id 产物；
- 清理前检查数据库引用；
- JSON 使用 UTF-8 和稳定 schema；
- 时间使用 ISO 8601。

## 13. 测试规范

- 新业务规则必须有单元测试；
- 修复缺陷先添加复现测试；
- 测试名称描述行为和期望；
- 不依赖测试执行顺序；
- 不访问真实网络；
- 不使用课程原始 CSV 做所有单元测试，应构造小夹具；
- 数据和模型集成测试可使用受控子集；
- 固定随机种子；
- 浮点比较使用明确容差；
- 慢测试标记；
- GPU 测试可跳过但必须有 CPU 等价路径。

详细矩阵见[测试与验收方案](10-test-and-acceptance.md)。

## 14. Git 工作流

### 14.1 当前状态

远程仓库为 [Timmyzzo/AI-Micro-Major-Project](https://github.com/Timmyzzo/AI-Micro-Major-Project)，默认分支为 main。main 必须与当前项目阶段一致并保持可审查；业务开发从聚焦的短分支开始，或在单人开发时以小步提交直接推进。

### 14.2 建议分支

- main：始终可演示；
- feat/需求编号-简述；
- fix/问题简述；
- docs/文档简述。

小型个人项目也可以直接使用短分支并频繁合并，避免长期大分支。

### 14.3 提交

- 一次提交一个逻辑；
- 每完成一个可独立验收的功能，在测试、文档和 diff 检查通过后立即提交并推送；
- 不把多个已完成的功能长期积压在工作区，也不把互不相关的功能合并为一个大提交；
- 不混入大型模型和缓存；
- 提交信息使用 feat、fix、docs、test、refactor、chore、data、model；
- 数据契约和模型结果提交中注明 dataset_id/run_id；
- 提交前检查 diff 和秘密。

### 14.4 忽略项

仓库 .gitignore 必须持续包含：

- .venv；
- .env；
- Python 缓存；
- IDE 本地设置；
- logs；
- data/interim、data/processed 的可再生产物；
- 大模型权重和缓存；
- artifacts 中临时导出；
- SQLite 临时日志；
- 用户上传文件。

课程要求必须提交的演示模型和截图可使用发布包、Git LFS 或单独网盘，不要直接塞入普通 Git 历史。

## 15. 代码审查清单

### 15.1 正确性

- [ ] 变更对应需求和验收条件；
- [ ] 时间边界正确；
- [ ] 单位正确；
- [ ] NaN、空集和除零处理正确；
- [ ] 不存在未来数据泄漏；
- [ ] 大模型没有承担数值计算；
- [ ] 缓存键包含所有影响结果的配置。

### 15.2 简洁性

- [ ] 没有多余服务、抽象和配置；
- [ ] 页面没有重复业务逻辑；
- [ ] 没有顺手重构无关代码；
- [ ] 公共依赖必要且有理由；
- [ ] 注释解释原因而不是复述代码。

### 15.3 安全

- [ ] 没有密钥；
- [ ] 路径经过限制；
- [ ] SQL 参数化；
- [ ] CSV 导出防公式注入；
- [ ] 日志脱敏；
- [ ] 上传内容不执行。

### 15.4 验证

- [ ] Ruff；
- [ ] mypy；
- [ ] pytest；
- [ ] 相关数据或模型测试；
- [ ] 手工页面检查；
- [ ] 文档更新。

## 16. AI 辅助开发规则

- 可以用 AI 起草代码、测试、文档和排错思路；
- 人工必须核对数据、单位、API 和运行结果；
- AI 生成的库用法应对照实际安装版本验证；
- 不把 AI 输出的虚构指标写进报告；
- 不把真实 API Key 放入提示词；
- 重要使用过程按学校要求记录；
- 最终责任由项目成员承担。

## 17. 完成定义

功能完成需同时满足：

1. 对应需求已实现；
2. 正常、边界和失败状态有测试；
3. 质量工具通过；
4. 数据和模型结果可复现；
5. UI 符合规格；
6. 失败可降级；
7. 文档和截图更新；
8. 无密钥和不必要产物；
9. 课程报告可引用真实证据；
10. 主分支可演示。
