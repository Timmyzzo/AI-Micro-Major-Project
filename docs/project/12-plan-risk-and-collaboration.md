- 基线与 PatchTST；
- 预测区间；

### Should

- 预警状态；
- 展示模式；
- 多方案比较；
- HTML 导出。

### Could

- 多变量 PatchTST；
- SHAP/注意力解释；
- Isolation Forest；
- Docker；
- 天气数据；
- 线性规划替代贪心。

### Won't for MVP

- 多用户；

当时间不足时，从 Could 开始删除，不降低数据正确性、测试和兜底。

## 7. 单人执行建议

如果由一人完成：

- 每日只推进一个主模块；
- 每个模块以测试或可见页面为完成条件；
- 每晚保存可演示版本；
- 模型训练在不阻塞编码时运行；
- 先形成完整丑版本，再统一美化；
- 报告图表从系统导出，不重复手工绘图；
- 答辩前三天停止新增功能。

建议看板列：

- Backlog；
- Ready；
- In Progress；
- Review/Test；
- Done；
- Deferred。

同时只能有一个高风险任务处于 In Progress，例如数据切分或主模型。

## 10. 缺陷优先级

| 等级 | 定义 | 示例 |
| --- | --- | --- |
| P0 阻断 | 无法演示或结果错误且无兜底 | 数据切分泄漏、应用无法启动、Key 泄漏 |
| P1 严重 | 核心功能错误但可绕过 | 预测导出错误、区间与时间错位 |
| P2 一般 | 次要功能或展示问题 | 某筛选无效、提示不清 |
| P3 低 | 轻微样式或改进 | 间距、非关键文案 |

## 12. 风险处理顺序

优先级：

1. 数据和方法正确性；
2. 密钥和安全；
3. 应用可启动与离线兜底；
4. 模型效果；
5. 页面美观；
6. 扩展创新。

漂亮界面不能抵消数据泄漏或虚构指标。

## 13. 变更管理

### 13.1 重大变更

以下需要正式记录：

- 改变预测目标、粒度、上下文或预测长度；
- 改变训练/验证/测试月份；
- 更换主模型；
- 引入新的外部数据；
- 引入真实控制或强化学习；
- 改为前后端分离；
- 改变 API 发送数据范围；
- 改变课程交付格式。

### 13.2 评估模板

每个重大变更回答：

1. 解决什么已存在的问题？
2. 是否有更简单方案？
3. 增加多少开发和测试工作？
4. 对截止时间和演示稳定性有何影响？
5. 哪些文档、数据、模型和测试失效？
6. 如何回滚？

没有明确收益和验收方式的变更不进入 MVP。
