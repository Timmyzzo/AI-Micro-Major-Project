# 01 项目立项与范围

## 1. 文档目的

本文档定义项目的背景、目标、范围、约束、交付物和成功标准。任何新功能都应先判断是否服务于本章目标，避免把课程项目扩展成难以完成的生产平台。

## 2. 项目名称

- 课程项目名称：电力数据智能分析与可视化系统
- 展示名称：智电洞察（PowerInsight）
- 基础实验：实验 6《家庭用电数据集的探索》
- 目标版本：课程演示 MVP 1.0

## 3. 背景与问题

基础实验完成了家庭用电数据的加载、描述统计、相关性分析和 Prophet 预测示例，但存在以下局限：

1. 预测示例随机抽取记录，破坏了时间顺序，不能作为严谨的多步时序预测流程。
2. 数据中的问号缺失值被简单转换为 0，会制造虚假的低负荷。
3. 没有训练集、验证集和测试集的时间隔离，也没有与基线模型比较。
4. 只展示单一预测图，没有数据质量、监测预警、决策支持和系统化界面。
5. 原说明将三个分项计量字段写成千瓦，UCI 原始定义实际为每分钟瓦时。
6. 没有记录模型、数据、参数、指标和报告之间的可追溯关系。

本项目不简单复刻实验，而是在保留同一数据主题的基础上，把它扩展成一个完整、可演示、可验证的智能分析系统。

## 4. 项目目标

### 4.1 总体目标

在一台配备 RTX 4060 Laptop GPU 的 Windows 游戏本上，完成一个界面美观、流程完整、技术路线较新且工作量可控的家庭电力数据分析系统。

### 4.2 业务目标

- 让使用者快速判断数据是否完整、可信。
- 直观了解家庭总负荷和分项用电的时间规律。
- 预测未来 24 小时负荷，并展示预测不确定性。
- 对数据缺失、异常波动和预测偏离进行分级预警。
- 模拟可转移负荷在峰谷时段之间调整后的影响。
- 用自然语言生成有证据、有限定条件的分析与建议。

### 4.3 教学目标

- 展示从数据治理到模型评估再到应用展示的完整机器学习工程流程。
- 使用 PatchTST、共形预测和大模型增强等相对前沿技术。
- 通过传统基线证明复杂模型是否带来真实改善。
- 体现对时间序列泄漏、模型不确定性和大模型幻觉的控制。
- 为学校小组项目报告提供图、表、指标、问题与解决方案的证据。

## 5. 用户与使用场景

### 5.1 主要用户

| 用户 | 诉求 |
| --- | --- |
| 课程答辩教师 | 快速理解功能闭环、技术创新、实验严谨性和实际结果 |
| 项目成员 | 导入数据、训练或加载模型、检查结果、准备截图和报告 |
| 演示操作员 | 在无网络或 API 异常时仍能稳定完成现场展示 |
| 家庭用能分析者 | 了解历史规律、未来高峰、异常和可行的节能情景 |

MVP 为本地单用户应用，不设计账号、组织、权限和多人并发。

### 5.2 关键场景

1. 用户使用内置 CSV，系统显示数据范围、质量和字段解释。
2. 用户查看某日、某周或整月的负荷与分项计量趋势。
3. 用户加载已训练 PatchTST，查看未来 24 小时预测和 90% 区间。
4. 用户启动历史回放，系统逐步显示观测值并触发异常提示。
5. 用户设定峰谷电价和可转移比例，比较调整前后的峰值与费用。
6. 用户调用兼容 OpenAI 接口生成总结；若 API 失败，仍获得模板报告。
7. 用户导出预测、预警或 Markdown 报告，用于课程材料。

## 6. MVP 范围

### 6.1 必须包含

| 能力域 | MVP 内容 |
| --- | --- |
| 数据采集 | 内置 CSV、上传 CSV、数据预览、字段映射与校验 |
| 数据治理 | 问号识别、缺失区间、重复与时间连续性、预处理摘要 |
| 数据监测 | KPI、趋势、分项构成、周期分析、模拟实时回放 |
| 负荷预测 | 基线模型、PatchTST、24 小时多步预测、区间和指标 |
| 预警 | 质量、规则、残差三类预警；等级、证据和导出 |
| 数据分析 | 峰谷、日周规律、相关性、典型日、用能构成 |
| 优化模拟 | 用户输入电价和转移比例，计算峰值与费用变化 |
| 大模型 | API 配置测试、结构化上下文、中文报告、失败回退 |
| 可视化 | 统一主题、交互式图表、状态提示、展示模式 |
| 可追溯 | 数据指纹、模型运行 ID、配置、指标和导出记录 |

### 6.2 应该包含

- 预测结果缓存，现场演示不必临时训练。
- 一键加载演示配置。
- Markdown 报告和 CSV 数据导出。
- 模型卡片，显示数据、时间切分、指标和限制。
- 异常确认状态，仅用于演示处置流程。

### 6.3 可以包含的扩展

- Isolation Forest 或自编码器异常分数，与残差预警对比。
- SHAP 或注意力可视化。
- Docker 打包。
- 外部天气数据。
- 真实流式消息或 MQTT 适配器。
- 小型强化学习负荷调度模拟环境。

扩展项不能影响 MVP 完成和演示稳定性。

## 7. 明确不做

- 不连接真实电表、电网、SCADA、PLC 或家庭设备。
- 不执行真实开关、调度、控制和计费。
- 不开发移动端、微信小程序或多租户平台。
- 不建设独立前后端、微服务、消息队列和分布式训练。
- 不进行用户登录、复杂权限和云端运维。
- 不将大模型用于核心数值预测。
- 不在缺乏动作、奖励和反事实数据时把强化学习包装成有效优化。
- 不声称系统告警等同于电气安全故障判断。

## 8. 约束与假设

### 8.1 已知约束

- 开发设备为 RTX 4060 游戏本，通常为 8 GB 显存，最终以实际设备为准。
- 数据只有一个家庭、六个月，不代表其他家庭或地区。
- 数据来自 2007 年，缺少天气、节假日、价格和住户行为标签。
- 用户可提供兼容 OpenAI 接口的 API，但提供商、模型名和可用性尚未锁定。
- 当前阶段不编写代码、不安装环境、不训练模型。
- 项目远程仓库为 https://github.com/Timmyzzo/AI-Micro-Major-Project，使用 main 作为默认分支。
- 后续每完成一个可独立验收的功能，都要及时提交并推送，避免功能积压和不可审查的大提交。

### 8.2 设计假设

- 课程展示以单机、单用户和离线数据为主。
- 15 分钟聚合能兼顾训练量、趋势可见性和本地资源。
- 用户可在演示前完成一次训练并保存最佳模型。
- 大模型调用允许只上传聚合指标和少量异常摘要，不上传完整原始数据。
- 峰谷电价由用户输入，系统不假设法国 2007 年实际电价。

## 9. 设计原则

1. **确定性计算优先**：统计、预测、异常分数和费用必须由程序计算，大模型只解释。
2. **时间顺序优先**：训练、验证和测试按月份切分，不随机打乱数据集边界。
3. **可展示但不造假**：界面强调技术亮点，同时明确假设、限制和待实测项。
4. **单体优先**：采用模块化 Streamlit 单体，减少接口、部署和联调成本。
5. **离线可用**：API 失败不阻断核心流程；训练和演示分离。
6. **基线优先**：主模型必须与简单模型比较。
7. **证据优先**：每个告警和建议都能追溯到时间、指标、阈值或模拟参数。
8. **原始数据不可变**：不覆盖课程提供的 CSV。

## 10. 成功标准

### 10.1 功能成功

- 从内置 CSV 到报告导出形成完整闭环。
- 所有 P0 功能均通过验收测试。
- 无 API 时，除大模型自由文本外的功能均正常。
- 页面在 1920×1080 投屏环境下无关键内容遮挡。

### 10.2 模型成功

- 使用固定的按月切分，不发生未来信息泄漏。
- PatchTST 在测试集主要指标上优于至少一个朴素基线；目标为 MAE 或 RMSE 相对改善不少于 5%。若未达到，必须如实报告并选择表现最好的可解释模型作为默认。
- 90% 预测区间在测试集的经验覆盖率目标为 85% 至 95%。
- 给出分预测步长和总体指标，而非只展示一张曲线。

### 10.3 工程成功

- 数据、模型、配置和结果具有版本或指纹。
- 单元、集成、数据和最小模型测试均可重复执行。
- 仓库中没有 API Key、个人信息和不必要的大型产物。
- 文档、界面和实际功能一致。

### 10.4 展示成功

- 8 至 10 分钟内完成一次稳定演示。
- 已缓存预测和报告可在网络失败时使用。
- 能清楚说明三个创新点：PatchTST、共形区间、证据约束的大模型解释。
- 能回答为何未使用强化学习、为何不能随机切分、为何不能把缺失填 0。

## 12. 变更控制

新增 P0 功能、改变数据切分、改变主模型、增加服务进程或扩大外部数据范围，均属于重大变更。重大变更必须同时更新需求、架构、测试、计划和风险文档，并确认不会影响课程截止时间。
