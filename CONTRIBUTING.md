# 项目协作与贡献指南

本文件是后续实现阶段的快速入口。详细规则以[开发规范](docs/project/09-development-standards.md)为准。

## 1. 开始工作前

1. 阅读 [README](README.md) 和[项目文档中心](docs/project/README.md)。
2. 确认需求编号、验收标准和受影响文档。
3. 检查当前 Git 状态、main 分支和远程仓库 [Timmyzzo/AI-Micro-Major-Project](https://github.com/Timmyzzo/AI-Micro-Major-Project)。
4. 不修改原始 CSV、原始实验说明和学校 Word 模板。
5. 不提交 API Key、个人信息、模型缓存、训练数据副本和大型模型权重。

## 2. 变更原则

- 每次变更只解决一个明确问题。
- 先写或更新验收条件，再实现。
- 优先复用既定模块，不增加未要求的服务和抽象。
- 数据处理必须按时间切分，禁止随机打乱后切分时序训练集和测试集。
- 模型结论必须与基线对比，不能只展示损失下降。
- 大模型只解释已计算证据，不得生成未经计算的电力数值。
- 对功能、配置、接口或数据契约的改动，必须同步更新对应文档。
- 每完成一个可独立验收的功能，就在检查通过后立即提交并推送，不把多个已完成功能积压成一个难以审查的大提交。

## 3. 计划中的本地质量检查

实现阶段应提供统一命令，至少执行：

~~~powershell
python -m ruff check .
python -m ruff format --check .
python -m mypy src
python -m pytest
~~~

涉及模型或数据时，还应执行数据校验、时间切分校验、基线评估和最小训练冒烟测试。以上命令目前仅是计划契约，依赖文件和代码尚未创建。

## 4. 提交节奏与提交信息

功能提交顺序：

1. 完成功能及失败状态；
2. 运行与该功能相称的测试和静态检查；
3. 更新需求、接口、使用说明或结果文档；
4. 检查 Git diff 和秘密；
5. 创建一个聚焦的提交；
6. 推送到远程仓库并确认远程提交存在；
7. 再开始下一个独立功能。

建议使用以下类型：

- feat：新增功能
- fix：修复缺陷
- docs：仅文档
- test：测试
- refactor：不改变行为的重构
- chore：构建、依赖或工具维护
- data：数据契约或数据处理变更
- model：模型配置、训练或评估变更

示例：**docs: 完成电力数据系统需求与架构基线**

## 5. 完成定义

一个变更只有在以下条件同时满足时才算完成：

- 需求与范围明确；
- 对应代码、测试和文档一致；
- 相关自动检查通过；
- 不泄露密钥或敏感信息；
- 结果可复现，配置和数据版本可追踪；
- 新功能有可操作的界面反馈和失败状态；
- 课程展示所需证据能够导出或截图；
- 已按功能边界提交并推送到远程仓库。
